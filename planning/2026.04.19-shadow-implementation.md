# Shadow Implementation Plan

## Background

In the iOS app, shadows are rendered via three mechanisms:

1. **Window inner shadows**: The `<window>` element's `shadowOpacity`, `shadowSigma`, and `shadowOffset` attributes control an **inner shadow** — a darkening effect along the inside edges of the window opening. This simulates the watch panel/bezel casting a shadow downward onto the recessed wheel beneath the window. The shadow is confined to the interior of the window and does not spread outward.

2. **Image-based shadows**: Henry uses ImageMagick `convert` at build time to generate shadow PNGs (e.g. `pm-window-border-shadow.png`) which are stored in the texture archive. Some watch faces reference these as `<Image>` parts. These are also inner-shadow effects, pre-rendered as bitmaps.

3. **Hand shadows**: In the iOS app, shadow hands track their parent hand — animating in sync, offset by XML-specified amounts. These are dynamic parts, not static images.

Since the web app has no Henry/ImageMagick pipeline, all shadows must be synthesized at runtime.

### XML attribute semantics

- **`shadowOpacity`** (0–1): Maximum darkness of the shadow at the edge. Controls `rgba(0,0,0,opacity)`.
- **`shadowSigma`**: Controls the blur radius / how far the shadow fades inward from the edge. Corresponds to a Gaussian blur sigma in ImageMagick's `convert`.
- **`shadowOffset`**: Vertical shift of the shadow. Simulates directional lighting:
  - `offset > 0` (e.g. Firenze, Miami, Chandra, Mauna Kea): Light from above — shadow is stronger on the **bottom** inner edge.
  - `offset < 0` (e.g. Terra, Gaia: `-0.5`): Light from below — shadow is stronger on the **top** inner edge.
  - `offset = 0` (e.g. Haleakala, Hana, Selene): Uniform shadow on all inner edges.

## Phase 1: Window Inner Shadows ✅ Complete

### Implementation

Window inner shadows are implemented in [renderer.ts](file:///Users/spucci/chronometer-web/src/watch/renderer.ts) using two functions:

- **`drawWindowBorder()`** — draws only the border stroke (no shadow logic)
- **`drawWindowInnerShadow()`** — draws Gaussian-curve gradient fills inside the window opening

#### Rendering pipeline integration

The inner shadow is drawn as part of the static cache, **after** the window hole is cut. This is critical because `cutWindowHole()` uses `destination-out` to erase the window interior — any shadow drawn before the hole would be wiped out.

The rendering order in all three call sites (`buildStaticBlockCaches`, `renderPartsWithWindows`, `renderWithWindowCutouts`) is:

1. `drawWindowBorder()` — stroke the border
2. `cutWindowHole()` — erase window interior to transparent
3. `drawWindowInnerShadow()` — paint semi-transparent gradients onto the transparent hole

When the static cache is composited onto the main canvas at frame time, the semi-transparent shadow pixels correctly darken whatever wheel content is showing through the window. **No per-frame cost.**

#### Gradient technique

For **rectangular windows**, four linear gradients (top, bottom, left, right) fade from `rgba(0,0,0,opacity)` at the edge to transparent inward. Each gradient uses `addGaussianStops()` which samples `e^(-x²/2)` at 8 points across 4σ, producing a smooth Gaussian-like falloff with no visible inner cutoff edge. The corners naturally receive double-darkening where gradients overlap, mimicking real shadow behavior.

For **porthole windows**, a radial gradient from the edge inward is used (same Gaussian stops). No porthole windows currently have shadow attributes, so this path is untested.

The `shadowOffset` parameter adjusts per-edge opacity multipliers: positive offset increases bottom-edge opacity and decreases top-edge (light from above); negative does the reverse.

#### Tuning

A global intensity multiplier of `0.5` is applied to all shadow opacities. This was tuned by visual comparison against the iOS app on Terra.

### Hana's `pm-window-border-shadow.png` — resolved

Hana previously used two `<Image>` parts referencing `pm-window-border-shadow.png` for the moonrise/moonset AM/PM windows. These were the only image-based shadows in the codebase. Fixed by adding `shadowOpacity='0.4' shadowSigma='1.5' shadowOffset='0'` directly to the two window elements and removing the orphaned `<Image>` parts.

## Phase 2: Hand Shadows (Future)

Hand shadows are more complex because they are **dynamic** — they rotate with their parent hand on every frame.

### How they work in iOS

- A shadow hand is a separate part in the archive that references the same angle expression as its parent hand
- It renders as a blurred, semi-transparent, offset copy of the parent hand's shape
- The offset is controlled by XML attributes (`shadowOffset`, etc.)
- Henry generates the shadow bitmap at build time, but the positioning/rotation is done at runtime

### Web approach (from prior discussion)

Add a `prerenderedBitmap?: ImageBitmap` field to the part model. At startup, for each part whose `src` matches a known shadow filename pattern, synthesize the bitmap on an `OffscreenCanvas`:

- Draw the parent hand's shape
- Apply Canvas 2D blur/shadow operations
- Convert to `ImageBitmap` and store on the part

The renderer would check `prerenderedBitmap` first; if present, blit it directly.

A `shadowSynthesizers` lookup table would map shadow PNG filenames to their synthesis algorithms.

### Memory budget

~0.16 MB per face (avg 10 shadows × 64×64 @ RGBA), ~4 MB for 25 faces — well within budget.

### Open questions

- **Which faces actually use hand shadows?** Need to audit the XML files for `<Image>` or `<QHand>` parts that reference shadow PNGs beyond `pm-window-border-shadow.png`.
- **Synthesis accuracy**: Canvas 2D `shadowBlur` may not perfectly reproduce ImageMagick's Gaussian blur, but a plausible approximation should be acceptable.
