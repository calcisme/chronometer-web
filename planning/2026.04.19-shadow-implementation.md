# Shadow Implementation Plan

## Background

In the iOS app, shadows are rendered via three mechanisms:

1. **Window inner shadows**: The `<window>` element's `shadowOpacity`, `shadowSigma`, and `shadowOffset` attributes control an **inner shadow** — a darkening effect along the inside edges of the window opening. This simulates the watch panel/bezel casting a shadow downward onto the recessed wheel beneath the window. The shadow is confined to the interior of the window and does not spread outward.

2. **Image-based shadows**: Henry uses ImageMagick `convert` at build time to generate shadow PNGs (e.g. `pm-window-border-shadow.png`) which are stored in the texture archive. Some watch faces reference these as `<Image>` parts. These are also inner-shadow effects, pre-rendered as bitmaps.

3. **Hand shadows**: Any hand with `z != 0` automatically gets a shadow generated by Henry. The shadow is a blurred, offset, semi-transparent copy of the hand's bitmap, rendered as a separate texture drawn immediately before the hand itself.

Since the web app has no Henry/ImageMagick pipeline, all shadows must be synthesized at runtime.

### XML attribute semantics (windows)

- **`shadowOpacity`** (0–1): Maximum darkness of the shadow at the edge.
- **`shadowSigma`**: Blur radius / how far the shadow fades inward.
- **`shadowOffset`**: Vertical offset (positive = light from above, stronger bottom edge).
- **`shadowOffsetX`**: Horizontal offset (negative = light from right, stronger left edge).

## Phase 1: Window Inner Shadows ✅ Complete

### Implementation

Window inner shadows are implemented in [renderer.ts](file:///Users/spucci/chronometer-web/src/watch/renderer.ts) using two functions:

- **`drawWindowBorder()`** — draws only the border stroke (no shadow logic)
- **`drawWindowInnerShadow()`** — draws Gaussian-curve gradient fills inside the window opening

#### Rendering pipeline integration

The inner shadow is drawn as part of the static cache, **after** the window hole is cut. This is critical because `cutWindowHole()` uses `destination-out` to erase the window interior — any shadow drawn before the hole would be wiped out.

The rendering order in all three call sites (`buildStaticBlockCaches`, `renderPartsWithWindows`, `renderWithWindowCutouts`) is:

1. `drawWindowBorder()` — stroke the border
2. `cutWindowHole()` — erase window interior to transparent
3. `drawWindowInnerShadow()` — paint semi-transparent gradients onto the transparent hole

When the static cache is composited onto the main canvas at frame time, the semi-transparent shadow pixels correctly darken whatever wheel content is showing through the window. **No per-frame cost.**

#### Gradient technique

For **rectangular windows**, four linear gradients (top, bottom, left, right) fade from `rgba(0,0,0,opacity)` at the edge to transparent inward. Each gradient uses `addGaussianStops()` which samples `e^(-x²/2)` at 8 points across 4σ, producing a smooth Gaussian-like falloff with no visible inner cutoff edge. The corners naturally receive double-darkening where gradients overlap, mimicking real shadow behavior.

For **porthole windows**, a radial gradient from the edge inward is used (same Gaussian stops). No porthole windows currently have shadow attributes, so this path is untested.

The `shadowOffset` parameter adjusts per-edge opacity multipliers: positive offset increases bottom-edge opacity and decreases top-edge (light from above); negative does the reverse. `shadowOffsetX` does the same for left/right.

#### Tuning

A global intensity multiplier of `0.5` is applied to all shadow opacities. This was tuned by visual comparison against the iOS app on Terra.

### Hana's `pm-window-border-shadow.png` — resolved

Hana previously used two `<Image>` parts referencing `pm-window-border-shadow.png` for the moonrise/moonset AM/PM windows. These were the only image-based shadows in the codebase. Fixed by adding `shadowOpacity='0.4' shadowSigma='1.5' shadowOffset='-1' shadowOffsetX='-1'` directly to the two window elements and removing the orphaned `<Image>` parts.

## Phase 2: Hand Shadows

Hand shadows are **dynamic** — they rotate with their parent hand on every frame and are offset to simulate height above the dial.

### How they work in iOS

#### Shadow generation (Henry / `makeOneShadow.pl`)

Any hand with `z != 0` automatically gets a shadow. Henry:
1. Renders the hand's bitmap to a PNG
2. Calls ImageMagick `convert` with `-background black -shadow {opacity}x{sigma}` to generate a blurred shadow PNG
3. Stores the shadow as a separate texture in the atlas, drawn immediately before the hand

#### Shadow parameters (from `makeOneShadow.pl`)

```
sigma = (z + 2) / 2

if (thick < 3.0):
    sigma *= thick / 3.0            # thinner hands → sharper shadows
    opacity = 50 + 50 * (3.0 - thick) / 3.0  # thinner hands → slightly darker (50→100%)
else:
    opacity = 50%
```

#### Shadow positioning (from `ECWatchController.m` lines 335-336)

The shadow is offset from the hand's anchor point:
- **X offset**: `+z / 4.3` (rightward — light from the left)
- **Y offset**: `-z / 2.15` (downward in screen coords — light from above)

Note: The Y direction in iOS screen coords is flipped relative to our XML coords.

#### Shadow drawing order

The shadow texture is rendered **immediately before** its parent hand, at z=0 (on the dial surface). It uses the same angle expression as the parent hand so it rotates in sync.

### Which hands have shadows

Every hand with `z != 0` in the XML gets a shadow. This is the vast majority of dynamic hands — hour, minute, second, azimuth, altitude, moonrise/set, etc. The `z` attribute represents the hand's "height" above the dial surface.

### Proposed web implementation

Since we don't have pre-rendered shadow bitmaps, we need to generate the shadow effect at runtime. Two approaches:

#### Approach A: Canvas shadowBlur on each hand draw (simpler, per-frame cost)

For each hand with `z > 0`:
1. Before drawing the hand, draw a "shadow copy" offset by `(z/4.3, z/2.15)` in XML units
2. Use Canvas `shadowBlur = sigma` and `shadowColor = rgba(0,0,0,opacity)` 
3. Or: draw the hand shape with a dark fill + opacity, then apply blur

**Issue**: Canvas `shadowBlur` may not play nice with the OffscreenCanvas contexts used for static caching. And per-frame blur is expensive.

#### Approach B: Pre-render shadow bitmaps at startup (matches iOS, zero per-frame cost)

At startup, for each hand with `z > 0`:
1. Draw the hand on a temp OffscreenCanvas using the same drawing code as the normal hand
2. Use CSS `filter: blur(sigma)` or Canvas `shadowBlur` to generate the blurred version
3. Tint to semi-transparent black
4. Store as an `ImageBitmap` on the part
5. At frame time, draw the shadow bitmap with the appropriate offset before drawing the hand

**Advantages**: Matches iOS exactly, zero per-frame blur cost
**Disadvantage**: More complex startup, some memory for cached bitmaps

#### Approach C: Use Canvas shadowBlur on the hand itself (simplest)

The simplest approach: when drawing a hand with `z > 0`, set Canvas shadow properties before the draw call:

```typescript
if (z > 0) {
    ctx.shadowColor = `rgba(0,0,0,${opacity})`;
    ctx.shadowBlur = sigma;
    ctx.shadowOffsetX = z / 4.3;
    ctx.shadowOffsetY = z / 2.15;
}
// draw hand normally — Canvas automatically renders the shadow
ctx.shadowColor = 'transparent';  // reset
```

**Advantages**: Almost no code changes needed — just add shadow setup before existing draw calls
**Disadvantage**: Canvas `shadowBlur` scaling/quality varies across browsers; per-frame cost for blur (but hands are small)

### Recommendation

Start with **Approach C** (Canvas shadowBlur on the hand draw) — it's by far the simplest and might be "good enough." If there are visual quality or performance issues, fall back to Approach B.

### Open questions

1. **Coordinate mapping**: The iOS offsets `z/4.3` and `z/2.15` are in screen pixels. We need to verify these map correctly to our XML-unit coordinate system given the scale factor.
2. **Image hands**: Hands with `src` attributes (PNG images like `moon25.png`) also have `z` and would need shadows too. Canvas `shadowBlur` should work on `drawImage()` as well.
3. **OffscreenCanvas support**: If hands are drawn on an OffscreenCanvas (e.g., for the Terra ring knockout cache), does `shadowBlur` work correctly there?
